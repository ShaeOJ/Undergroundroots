The Concept

A mining cluster where every node is an ESP32-S3, the master holds the pool connection, and all coordination happens over ESP-NOW — a low-latency 802.11 broadcast protocol that requires no access point. The NerdQAxe++ acts as master: it connects to the stratum pool, generates work, and broadcasts jobs to up to 8 slave devices. Slaves mine, find shares, and report back. The pool sees one worker. The cluster acts as one miner.

Architecture

The master runs the full stratum session — mining.subscribe, mining.authorize, job management, share submission. When a new job arrives from the pool, the master carves the 32-bit nonce space into equal slices and broadcasts a $CLWRK message to each registered slave, assigning each node a unique nonce range. With 5 nodes (1 master + 4 slaves), each covers exactly 20% of the search space.

Slaves register on boot by broadcasting a $REGISTER beacon. The master responds with $CLACK containing the slave's assigned ID and nonce slot. Heartbeats flow every 3 seconds — if a slave misses 15 consecutive heartbeats (30 seconds), the master drops it. Slaves self-deregister after 35 seconds without work, then re-register automatically.

The Bugs That Almost Broke It

Getting 8 nodes to stay alive simultaneously took several rounds of fixes. The major ones:

RX Queue Overflow

ESP-NOW traffic (beacons, work broadcasts, heartbeats, share reports) all flows through a single queue. With 8 slaves, work broadcasts arrive for all nodes simultaneously and filled the 16-slot queue, causing heartbeats to be silently dropped. The master's timeout fired, dropped the slave, and the slave re-registered — a constant churn that looked like instability. Fix: doubled the RX queue to 32 slots on both master and slave firmware.

ASIC Queue Idle

After the stability fix, slaves showed dramatically fewer shares. The root cause: cluster_slave_submit_to_asic enqueued exactly one job when work arrived, then the ASIC processed it in ~700ms and called queue_dequeue — which blocked forever on pthread_cond_wait. The slave was only mining for 700ms per master broadcast, then sitting idle. The fix added a refill loop: when the job queue drops below watermark, the slave reconstructs a job from the last received work and enqueues it, keeping the ASIC continuously fed.

Stale Job Accumulation

Work arrives from the master every ~500ms. Without clearing the queue on new work, stale jobs accumulated over 32 slots. The ASIC was mining old templates and submitting shares the pool rejected as stale. One line fix: ASIC_jobs_queue_clear() before every new enqueue — the slave always mines the freshest work.

Full Cluster Rejection

When a cluster is full (8 slaves registered), the master sends $CLACK,0,FULL. The original slave code set registered = true unconditionally — a rejected slave would claim slot 0 (the master's own slot), send heartbeats on it, and never receive work. Fixed with an explicit rejection check before marking registered.

Auto-Timing Calibration

Each node runs an auto-timing calibrator that finds the optimal ASIC job submission interval for its silicon. The calibration sweeps 10 interval steps from 350ms to 750ms, spending 180 seconds at each step to gather reliable share statistics. The winner is saved to NVS flash and restored on every reboot — a fresh calibration only runs if the device has never been calibrated, or if the user triggers it manually from the Tuner page.

The Tuner UI

ClusterAxe's web UI is built in Angular with PrimeNG components. The Tuner page exposes the watchdog (voltage throttle and recovery thresholds) and the auto-timing calibration state. The benchmark tool runs a pure-frontend frequency and voltage sweep using the existing system API — no new firmware endpoints. It logs hashrate, efficiency (J/TH), and hardware error rate per test point, identifies the best stable operating point, and writes it to NVS immediately whenever a new best is found mid-run.
